# Loopit 创作者体系：项目关注重点与 MVP 建议

## 1. 背景与项目判断

本次会议讨论的创作者体系包含年龄与动机分层、成长路径、任务、Level、权益、挑战、奖励和主动运营触达等内容。

这意味着当前项目的目标正在从“作品数据诊断助手”扩展为“创作者成长运营 Agent”。两者的复杂度差异很大：当前项目主要帮助创作者查询作品数据、理解玩家反馈并获得优化建议；完整创作者体系还需要可靠地管理用户状态、任务进度、等级变化、奖励发放和运营触达。

项目需要坚持以下核心边界：

> Agent 不是创作者体系本身，而是创作者体系的沟通与个性化执行层。

后台系统负责保存事实、执行规则和产生状态变化；Agent 负责问卷沟通、解释规则、推荐下一步、生成反馈和提醒用户。

## 2. 创作者体系建议架构

创作者体系至少需要表达以下关系：

```text
Creator
 ├─ Age band（年龄区间）
 ├─ Contact permission（官方联系权限）
 ├─ Motivation tags（创作动机标签）
 ├─ Growth path（成长路径）
 ├─ Current level（当前等级）
 ├─ Active tasks（进行中的任务）
 ├─ Completed challenges（已完成挑战）
 ├─ Granted rewards（已获得奖励）
 └─ Operation history（历史运营动作）
```

推荐的用户运行流程：

```text
进入创作者体系
  → 确认年龄区间与联系权限
  → 完成结构化动机问卷
  → 后台确定成长路径
  → 解锁对应任务
  → 用户完成任务
  → 后台校验任务结果
  → 更新 Level
  → 发放或审批权益
  → Agent 解释结果并推荐下一步
```

## 3. Agent 与后台的职责边界

### 3.1 后台系统负责

- 用户身份、年龄区间与联系权限。
- 用户标签和分群计算。
- 成长路径、Level、任务和挑战的结构化配置。
- 任务完成条件与数据口径。
- 用户任务进度和 Level 状态。
- 奖励资格计算、发放、过期与回收。
- 触达频率、静默期和渠道限制。
- 运营动作历史、消息状态和用户反馈。
- 所有写操作的权限、幂等与审计。

### 3.2 Agent 负责

- 通过对话完成问卷，但将答案写入结构化字段。
- 根据后台返回的标签、路径和任务解释用户当前状态。
- 推荐最合适的下一项任务或活动。
- 结合作品数据和玩家评论给出个性化建议。
- 根据后台事件生成提醒、鼓励和阶段总结。
- 在触达前参考历史运营动作，避免重复表达。
- 当信息不足或规则不允许时停止行动并说明原因。

### 3.3 Agent 不应负责

- 根据语言、作品或聊天表现猜测用户年龄。
- 自行决定用户属于哪个正式年龄层或权益层。
- 仅根据 Prompt 临时计算 Level。
- 自行认定任务完成并直接发放高价值奖励。
- 绕过后台频控直接发送消息。
- 访问通用 SQL 或任意修改后台数据。

## 4. 项目最值得关注的问题

### 4.1 年龄与联系权限是前置条件

会议提到大量创作者可能低于 13 岁。年龄区间会直接影响可进入的成长路线、可参加的活动、允许收集的信息、是否允许个性化运营以及官方触达方式。

建议后台至少提供：

```text
age_band: under_13 | 13_to_17 | adult | unknown
contact_permission: allowed | restricted | denied
```

Agent 只能使用后台确认的状态，不能自行推断。`unknown` 应进入经过产品和相关负责人确认的保守路线。

### 4.2 用户分层必须可执行

当前项目中的用户分层只是名称和自然语言描述。完整体系需要结构化、可计算的规则，例如：

```json
{
  "segmentId": "new_creator_13_plus",
  "ageBands": ["13_to_17", "adult"],
  "contactPermission": "allowed",
  "conditions": {
    "publishedWorks": 0,
    "registeredWithinDays": 7
  },
  "growthPathId": "first_publish"
}
```

运营负责定义目标人群和规则，后台负责计算分群结果，Agent 只读取确定的结果。

### 4.3 成长路径需要正式状态模型

需要在开发前确认：

- 一个用户能否同时进入多条成长路径。
- 创作动机变化后能否切换路径。
- Level 是全局等级还是每条路径独立等级。
- 任务是否有前置条件、有效期和重试次数。
- Level 是否会降低或重置。
- 活动挑战是否会影响 Level。
- Playpass、Growpass、Coobra Program 是成长路线、权益包还是用户类型。

这些规则不应写在 Agent Prompt 中，应形成后台可版本化的配置和状态机。

### 4.4 调度应以业务事件为主

完整体系不适合每天让模型重新扫描所有用户并判断发生了什么。推荐由后台事件触发 Agent：

```text
用户注册 / 问卷完成 / 路线确定 / 任务解锁 / 作品发布
任务完成 / Level 变化 / 奖励发放 / 活动开始或结束
  → 触发对应的 Agent 沟通任务
```

定时任务只用于任务到期提醒、阶段总结、长期未行动召回和周期性作品复盘。

### 4.5 避免重复需要运营动作账本

只依赖 Agent 会话记忆无法可靠避免重复触达。后台需要保存结构化运营记录：

```text
user_id
action_type
growth_path_id
task_id
campaign_id
message_template
sent_at
delivery_status
user_response
cooldown_until
```

Agent 在生成内容前读取相关历史；后台在发送前再次执行频控和去重。

### 4.6 奖励属于高风险写操作

称号、头像框、积分、流量、加热券和实物奖励需要统一处理：

- 资格计算和任务完成证据。
- 幂等键，防止重复发放。
- 数量限制和库存。
- 生效、过期和回收时间。
- 异常指标和作弊处理。
- 发放失败后的重试或补偿。
- 人工审批和申诉记录。

MVP 阶段建议由 Agent 生成奖励建议，后台规则或运营人工确认后再执行发放。

### 4.7 运营配置需要结构化

创作指导、案例解析和活动介绍可以继续使用 Markdown 知识库。以下内容则必须结构化并支持版本管理：

- 成长路径和 Level 规则。
- 任务与挑战定义。
- 奖励目录和发放条件。
- 用户分层和年龄限制。
- 频控规则。
- 活动起止时间和参与条件。

## 5. 当前项目能力的复用与缺口

### 5.1 可以复用

- 创作者聊天入口。
- UID/PID 作品查询工具。
- 数据表现、评论和创作历程诊断能力。
- 创作指导与运营活动知识库。
- 交互会话与主动触达会话隔离。
- `NO_OUTREACH` 判断机制。
- outbox 人工审核的基本思路。
- 流式展示回复和工具调用过程。

### 5.2 需要接入正式后台能力

- 用户身份、年龄区间与联系权限。
- 结构化标签和真实用户分群。
- 成长路径、Level 和任务进度。
- 挑战参与和完成状态。
- 奖励资格、审批与发放。
- 真实 IM/Push 发送通道。
- 运营动作历史和强制频控。
- 正式数据库和可靠任务调度平台。

### 5.3 当前明显断点

- 页面中的用户分层只有文字配置，不能真正圈选用户。
- 页面中的定时任务配置尚未接入运行时调度器。
- “立即试发”只是发送到本地测试聊天。
- outbox 的 deliver 只更新本地状态，没有调用真实消息系统。
- 会话、任务和运行记录使用本地 JSON，不适合生产多实例。
- 当前数据工具以作品诊断为主，尚无成长体系相关后台工具。

## 6. MCP 接口建议

如果用户、任务、奖励、活动和消息能力会被多个 Agent 或应用复用，可以由后台服务通过 MCP 暴露业务工具。MCP 是连接协议，不负责替代数据库、规则引擎、鉴权、调度器或消息队列。

### 6.1 只读工具

```text
get_creator_profile
get_creator_segment
get_contact_policy
get_growth_path
list_active_tasks
get_task_progress
list_eligible_challenges
list_available_rewards
get_operation_history
```

### 6.2 有副作用的工具

```text
record_questionnaire_answers
enroll_growth_path
assign_creator_task
claim_task_completion
grant_reward
send_creator_message
```

写操作必须具备：

- 服务端权限校验。
- 结构化参数校验。
- 幂等键。
- 操作审计。
- 必要时的人工审批。
- 明确的错误码和重试策略。

不建议向 Agent 暴露通用 `execute_sql` 或任意后台写入工具。

## 7. 推荐的最小 MVP

第一版不同时实现三条成长路线、完整等级体系、积分商城、头像框、称号和流量奖励。建议只验证一条真实闭环：

- 一个年龄状态和联系权限已经确认的人群。
- 一条成长路径。
- 2～3 个完成条件明确的任务。
- 2 个 Level。
- 一种低风险奖励。
- 两类触发事件：加入路线、完成任务。
- Agent 负责解释、提醒和反馈。
- 消息与奖励先经过人工审核。

示例：

```text
目标人群：符合联系条件的新创作者
成长目标：完成首次发布
L0：尚未发布
L1：完成首次发布
任务：形成想法 → 创建作品 → 发布作品
奖励：新手称号或固定轻权益
Agent：引导任务、适度提醒、完成后反馈并推荐下一步
```

MVP 的目的不是证明完整体系可以配置，而是验证：

1. 后台能准确确定用户状态和下一任务。
2. Agent 能把任务和权益解释清楚。
3. 触达不会重复或打扰用户。
4. 用户完成目标的比例得到改善。
5. 运营可以看见并复盘每次 Agent 行为。

## 8. 运营与产品职责建议

### 8.1 运营负责

- 设计成长路线、任务内容和活动玩法。
- 定义目标分群和触达策略。
- 规划权益包与奖励稀缺性。
- 维护创作指导、案例和活动内容。
- 审核高价值奖励与重要曝光。
- 复盘任务完成率、触达效果和用户反馈。

### 8.2 产品与研发负责

- 设计统一的数据模型和状态流转。
- 提供身份、分群、任务、奖励和消息服务。
- 明确任务完成口径和事件来源。
- 建设权限、幂等、频控和审计机制。
- 为 Agent 提供稳定的业务 API 或 MCP 工具。
- 确保客户端对 Level、称号、头像框等权益的正确展示。

### 8.3 共同负责

- 罗列并定义所有挑战类型。
- 确认每种挑战的完成证据。
- 确认每条成长路径的目标和成功指标。
- 确定奖励价值、数量、有效期和回收策略。
- 确认不同年龄区间可使用的运营路线。

## 9. 下次会议前需要确认的问题

1. Playpass、Growpass、Coobra Program 的准确定义和相互关系。
2. 年龄区间划分，以及每个区间允许的运营动作。
3. 用户如何确认或更新创作动机。
4. Level 是全局等级还是成长路径等级。
5. 任务、挑战和活动三者的区别。
6. 每类任务的完成证据和数据口径。
7. 用户能否同时进入多条成长路径。
8. 奖励由规则自动发放还是运营审批。
9. 称号、头像框、流量权益的过期和回收方式。
10. 用户如何退出体系或关闭官方触达。
11. 运营配置范围与产品后台强约束范围。
12. 第一条试点成长路线、目标用户和成功指标。

## 10. 最终建议

当前项目应继续保持小 MVP，不要把完整创作者体系规则塞进 Agent Prompt。

近期开发的正确顺序是：

```text
明确一条试点成长路径
  → 定义后台状态和任务口径
  → 接入真实身份与事件
  → Agent 解释和触达
  → 人工审核消息与奖励
  → 记录效果并复盘
  → 再逐步扩展更多路线和自动化能力
```

项目只应自行实现 Loopit 特有的成长策略和沟通逻辑。身份、存储、调度、分群、消息发送、观测和权限应优先复用公司现有服务或成熟第三方能力。
